# 短视频文案系统 MVP 补充计划

本文档补充《短视频文案系统_MVP方案_内部版.md》，用于把前期方案收敛为第一版可执行计划。当前补充计划不替代原方案，后续可以在实现前合并回主方案文档。

## 1. MVP 定位

第一版系统定位为内部团队的短视频创作参考工具，主要服务内容运营和编导。

系统处理团队成员主动提交的抖音单条公开视频链接，或本地上传的视频/音频文件，生成转写稿、清洗稿、结构拆解和改写版本。所有输出只作为内部创作参考，不定位为自动洗稿工具，也不承诺生成内容可以直接发布。

发布前必须人工确认，尤其要检查事实性内容、品牌信息、广告极限词、版权风险和平台规则风险。

第一版不做多用户权限、主页采集、合集采集、批量采集、字幕时间线编辑、剪映格式导出和复杂审批流程。

## 2. 工程路线

当前仓库作为最终项目仓库使用。第一版采用一次性导入 `AI-Video-Transcriber` 的方式建设主项目底座，不以持续同步上游更新为目标。

`AI-Video-Transcriber` 作为 Web、上传、转写、文本优化、Docker 部署等能力的启动模板。导入后，以本项目的短视频文案工作台需求为主进行二开，上游非核心功能保留但在产品入口上弱化。

抖音解析与下载能力参考高 star 项目 `Evil0ctal/Douyin_TikTok_Download_API` 的架构思路，重点借鉴：

- `FastAPI + crawlers + config` 的组织方式。
- 通过 Cookie 提升抖音公开视频解析和下载成功率。
- Docker 部署下的配置文件和运行依赖组织方式。

原方案中的 `jiji262/douyin-downloader` 不再作为主选下载底座，可作为后续备选或对比参考。

## 3. 抖音下载与 Cookie 策略

第一版能力边界定义为：支持当前部署环境可访问、且配置 Cookie 后可解析的单条抖音公开视频链接。

系统不承诺处理以下内容：

- 私密视频。
- 已删除视频。
- 当前 Cookie 对应账号无权限访问的视频。
- 被平台风控、验证码或地区策略阻断的视频。
- 主页、合集、搜索页和批量链接。

抖音 Cookie 不在 Web 页面中填写。部署时通过环境变量注入，并落地到容器内配置文件，供下载解析模块读取。Docker 镜像中不得内置个人 Cookie。

如果抖音链接下载失败，用户可以上传本地视频或音频作为兜底输入，继续走转写、清洗、结构拆解和改写链路。

## 4. 任务链路

提交任务后，系统立即创建任务记录并返回任务 ID。后台使用单 worker 串行执行任务，避免本机 Docker 环境下多个 Whisper/下载任务并发导致资源打满。

任务状态建议包含：

- `pending`：已创建，等待处理。
- `downloading`：正在解析和下载抖音视频。
- `extracting_audio`：正在转音频。
- `transcribing`：正在语音转写。
- `cleaning`：正在清洗、加标点和分段。
- `analyzing`：正在生成简版结构拆解。
- `rewriting`：正在生成改写结果。
- `completed`：任务完成。
- `failed`：任务失败。

为支持按步骤重试，关键中间产物需要持久保存：

- 原始视频。
- 提取后的音频。
- 原始转写稿。
- 人工修订后的转写稿。
- 清洗稿。
- 结构拆解。
- 改写结果。
- 导出文件。
- 错误信息。

失败后允许按步骤重试。容器重启时，处于运行中状态的任务统一标记为失败，由用户在页面上手动重试。

## 5. 文案处理流程

第一版采用两步 LLM 流程。

第一步：基于转写稿生成清洗稿，完成去口水词、加标点、分段和基础可读性整理。

第二步：基于人工确认后的转写稿和清洗稿生成结构拆解与改写结果。

任务详情页需要允许用户编辑转写稿。后续清洗、结构拆解和改写应优先使用人工修订后的转写稿，而不是原始 Whisper 输出。

结构拆解采用简版输出，包含：

- 开头钩子。
- 核心卖点。
- 节奏转折。
- 结尾行动。

默认生成三类改写结果：

- 口播稿。
- 种草笔记文案。
- 标题和开头钩子。

## 6. 页面能力

第一版 Web 最小页面包含四类。

提交页：

- 输入抖音单条视频链接。
- 上传本地视频或音频作为兜底。
- 使用环境变量中的默认 LLM 配置。
- 保留上游 UI 覆盖 LLM 配置能力。

任务列表页：

- 按时间倒序展示历史任务。
- 支持按视频标题、原链接和任务状态搜索。
- 展示任务状态、输入类型、创建时间、完成时间和错误状态。
- 不记录提交人，不按用户筛选。

任务详情页：

- 展示来源链接、视频标题、作者昵称和采集时间。
- 展示并编辑转写稿。
- 展示清洗稿、结构拆解和改写结果。
- 支持 Markdown 导出、Docx 导出和一键复制。
- 每个改写结果支持“好用 / 不好用 + 原因标签”反馈。
- 支持失败步骤重试。

统计页：

- 展示任务总数。
- 展示任务成功率。
- 展示平均完成耗时。
- 展示反馈好评率。

共享口令用户均可访问统计页，第一版不做管理员角色。

## 7. 配置与部署

Python 依赖与环境统一使用 uv 管理（`uv sync` / `uv add --dev` / `uv run`），不使用裸 pip 或手动 venv。代码质量与架构检查逻辑集中在 `scripts/checks/`，作为 Claude Code hooks、git pre-commit 与 Codex 共用的单一事实源，具体见根目录 `AGENTS.md`。

第一版本机 Docker 部署优先，但目录、环境变量和镜像结构需要兼容后续迁移到单台服务器。

Docker Compose 第一版使用单应用容器。FastAPI/Web、抖音解析下载、FFmpeg、faster-whisper、LLM 调用和单 worker 后台任务都在同一个容器内运行。

数据目录挂载到项目下的 `data/`，用于保存：

- SQLite 数据库。
- 原始媒体文件。
- 转写和文案中间产物。
- 导出文件。
- 模型缓存。
- 下载器配置文件。

`data/` 不应提交到 Git。

访问控制使用团队共享口令。第一版不做账号体系、角色权限和个人任务归属。

LLM 配置策略：

- 环境变量作为默认配置，支持 OpenAI-compatible API。
- 保留上游 UI 覆盖配置能力。
- UI 输入的 API Key 不在服务端持久化。

faster-whisper 默认使用 `small` 级别模型，优先平衡中文识别质量、速度和本机资源占用。

## 8. 文件保留策略

任务完成后，原始视频和音频默认保留 7 天，便于排查问题和支持近期重试。

7 天后自动清理原始视频和音频，但保留：

- 原始转写稿。
- 人工修订稿。
- 清洗稿。
- 结构拆解。
- 改写结果。
- 导出文件。
- 反馈记录。
- 任务元信息。

媒体清理后，任务仍允许重跑文本阶段，包括清洗、结构拆解和改写。若需要重新下载或重新转写，用户需要重新提交链接或上传本地文件。

## 9. 成功指标

第一版通过系统数据和人工反馈共同评估。

核心指标：

- 从提交到完成的平均耗时。
- 任务成功率。
- 改写结果好评率。
- 团队每周实际任务数。

统计页展示任务数、成功率、平均耗时和反馈好评率。每两周复盘一次，判断下一步优先修稳定性、转写质量、改写质量还是使用体验。

## 11. 验收标准

第一版可按以下场景验收：

- Cookie 配置正确时，单条可访问抖音公开视频能完成下载、转写、清洗、结构拆解、改写和导出。
- 抖音下载失败时，本地上传兜底链路能完成同样处理。
- 转写稿编辑后，后续清洗和改写使用人工修订稿。
- 任一步失败后，任务进入失败状态，并支持从对应步骤重试。
- 容器重启后，运行中任务被标记失败，不出现永久卡住状态。
- 7 天媒体清理不删除文本结果、导出文件和反馈记录。
- 统计页能展示任务数、成功率、平均耗时和反馈好评率。
